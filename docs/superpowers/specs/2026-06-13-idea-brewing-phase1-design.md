# idea brewing — 第1版(レシピ醸造)設計書

- 日付: 2026-06-13
- ステータス: 承認済み(設計)
- スコープ: 第1版「原料投入 → 仕込み → グリル → レシピ生成」まで

## 1. サービス概要

idea brewing は、ユーザーが投入したテキスト・画像・URL・ファイルを原料として、設定された LLM を使い 1 つのサービスへと「醸造」していくローカルアプリケーションである。ビール醸造のメタファーでアイデアの精緻化プロセスを表現する。

「最高の todo アプリ」という一言でも、参考 LP の画像や URL でも、詳細な計画書ファイルでも、それらの組み合わせでも原料として受け付ける。雑な原料から始めても、グリル工程(質問攻め)を通じて実装可能な水準まで煮詰められることが本サービスの中核価値である。

### 全体パイプラインと本設計の範囲

| 醸造工程 | アプリ上の意味 | フェーズ |
|---|---|---|
| 原料投入 | テキスト・画像・URL・ファイルの投入 | **第1版** |
| 仕込み(マッシュ) | LLM が原料をブリューシートに正規化 | **第1版** |
| 煮沸・ホップ投入(グリル) | grill-me-with-docs 方式の質問で曖昧さを煮詰める | **第1版** |
| 発酵 | 実装資料(レシピ)一式の生成 | **第1版** |
| ビルド | レシピを Cursor CLI/SDK に渡してコード生成 | Phase 2 |
| 熟成・バッチ | 自己評価 → 自己改善のループ(1st/2nd バッチ…) | Phase 3 |
| Pub | AI 客によるユーザーテスト・リーダーボード | Phase 4 |

## 2. 確定済みの前提

- **形態**: ローカル起動の Web アプリ。ユーザー自身の PC で起動しブラウザで使う。データはすべてローカル保存。
- **エンジン方針**: ハイブリッド。資料生成・評価は BYOK の LLM API を内蔵呼び出し。コード生成(Phase 2 以降)は Cursor CLI / Cursor SDK に委譲する。
- **LLM 接続先(初期必須)**: OpenAI / Google(Gemini)/ Ollama(OpenAI 互換ローカル)/ OpenRouter。
- **醸造対象の成果物**: Web アプリ / LP に限定(ローカルで dev サーバー起動してブラウザで動くもの)。
- **技術スタック**: Next.js(App Router・TypeScript・Tailwind CSS)単一アプリ + Vercel AI SDK + ファイルベース保存。

## 3. UI コンセプト

- 銅・琥珀色を基調とした醸造所ダッシュボード。背景・アクセントに泡や液面のモチーフを使う。
- 1 案件 = 1 ブリュー(brew)。ダッシュボードには各ブリューが「醸造タンク」のカードとして並ぶ。
- タンクの液面の高さと泡のアニメーションで工程進捗(原料投入 → 仕込み → グリル → レシピ完成)を表現する。
- ブリュー詳細画面は工程ステッパー(原料 / 仕込み / グリル / レシピ)で構成し、現在工程が常に分かる。
- UI 言語は日本語。

## 4. ブリューシート(アイデア構造化フレームワーク)

雑な原料を受け止める器として、7 項目の「ブリューシート」を定義する。仕込み工程で LLM が原料から各項目を埋める。

1. **コンセプト** — 何を・誰のために・なぜ作るのか
2. **ターゲットユーザーとコアジョブ** — 想定ユーザー像と、解決する中心課題
3. **主要機能** — Must / Should / Could の 3 段階で列挙
4. **見た目とトーン** — 参考画像・URL からの抽出を含むビジュアル方針
5. **成功基準** — 何ができたら「最高」と言えるかの判定条件
6. **制約** — 技術制約・スコープ外事項
7. **自己評価の観点** — Phase 3 の評価ループで使う採点軸の種

各項目は**充足度(充足 / 薄い / 空)**を持つ。LLM が仕込み時に判定し、グリル工程の質問対象選定に使う。ユーザーはシートをいつでも直接編集でき、編集すると充足度は再判定される。

- 「最高の todo アプリ」の一言 → ほぼ全項目「薄い/空」から開始し、グリルで埋める。
- 詳細な計画書ファイル → 最初からほぼ「充足」で開始し、グリルは短く済む。

## 5. 機能仕様

### 5.1 原料投入

- 入力手段: フリーテキスト、URL(複数可)、画像ファイル、ドキュメントファイル(.md / .txt / .pdf)。ドラッグ & ドロップ対応。
- 原料は組み合わせ自由。投入後も(レシピ生成前なら)追加投入できる。追加時は仕込みを再実行してブリューシートに反映する。
- URL はサーバー側で fetch し本文を抽出してテキスト原料化する。取得失敗は「取り込み失敗」として原料リストに記録し、醸造自体は継続できる。
- 画像は保存し、ビジョン対応モデル設定時は仕込みのマルチモーダル入力として使う。非対応モデルの場合は「画像は参照されない」旨を UI に明示する。
- PDF はテキスト抽出して原料化する。

### 5.2 仕込み(マッシュ)

- 「仕込み開始」操作で、全原料を入力として LLM がブリューシート 7 項目を生成し、項目ごとの充足度を判定する。
- 結果はブリューシート編集画面に表示。ユーザーは手動修正可能。
- 仕込みは何度でも再実行できる(原料追加後など)。再実行時はユーザーの手動編集内容を優先し、上書きしない(LLM には「ユーザー確定済み」として渡す)。

### 5.3 グリル(grill-me-with-docs)

- 充足度の低い項目・項目間の矛盾・曖昧な表現を LLM が検出し、**1 問ずつ**質問を生成する。
- 各質問は 2〜4 個の選択肢を持ち、推奨選択肢にはマークを付ける。自由記述回答も常に可能。
- 回答するたびにブリューシートが更新され、画面上の充足度ゲージが上がる。
- **auto モード**: ON にすると推奨選択肢が自動採用されて連続進行する。ユーザーはいつでも一時停止して手動回答に切り替え、また質問履歴を遡って回答をやり直せる(やり直した場合、それ以降の回答とシート反映は破棄して再生成)。
- 終了条件: 全項目が「充足」になるか、ユーザーが「煮詰め完了」ボタンを押す。LLM 側にも「これ以上聞くことがない」判定をさせ、上限(デフォルト 20 問)で必ず終了する。
- 質問と回答の全履歴は保存し、ブリュー詳細から参照できる。

### 5.4 レシピ生成(発酵)

確定したブリューシートとグリル履歴から、以下の実装資料一式を Markdown で生成し `recipe/` に保存する。

| ファイル | 内容 |
|---|---|
| `00-overview.md` | サービス概要・エレベーターピッチ |
| `01-requirements.md` | 要件定義(機能要件・非機能要件) |
| `02-screens.md` | 画面一覧・画面ごとの構成・UX フロー |
| `03-design-system.md` | UI/UX デザイン指針(配色・タイポグラフィ・コンポーネント方針・参考ビジュアルの言語化) |
| `04-architecture.md` | 技術構成(スタック・データモデル・ディレクトリ構成) |
| `05-implementation-plan.md` | 実装計画(Phase 2 でそのまま Cursor CLI に渡せる粒度のタスク分解) |
| `06-evaluation-criteria.md` | 自己評価基準(採点表。Phase 3 の評価ループの入力) |

- 生成は 1 ファイルずつ順番に行い、進捗をタンク UI に反映する。
- 生成後も「再発酵」(ブリューシート変更後の再生成)が可能。再生成時は旧版を `recipe/history/<タイムスタンプ>/` に退避する。
- `06-evaluation-criteria.md` はブリューシートの「成功基準」「自己評価の観点」を、採点可能なルーブリック(観点 × 5 段階基準)に展開したものとする。これが Phase 3 の自己評価ループとの契約書になる。

### 5.5 設定(BYOK / ローカル LLM)

- プロバイダ登録: OpenAI / Gemini / OpenRouter / Ollama(OpenAI 互換エンドポイント)。API キー・ベース URL(Ollama 等)・モデル名を設定。
- 接続テストボタン(小さなプロンプトを投げて応答確認)。
- 第1版では「既定モデル」を 1 つ選ぶ方式とする(工程別のモデル使い分けは将来拡張)。
- キーはローカルの `data/settings.json` にのみ保存し、外部送信はプロバイダ API 呼び出し以外で行わない。`data/` は `.gitignore` 対象。

## 6. アーキテクチャ

### 6.1 構成

- Next.js(App Router)単一アプリ。UI と API(Route Handlers)を同一プロセスで提供。`npm run dev` で localhost 起動。
- LLM 層は Vercel AI SDK で抽象化。`@ai-sdk/openai` / `@ai-sdk/google` / OpenAI 互換プロバイダ(Ollama・OpenRouter)をアダプタ登録し、工程ロジックからはプロバイダ非依存のインターフェースで呼ぶ。
- グリルの質問生成・仕込みのシート生成は構造化出力(JSON スキーマ)で受け取る。レシピ生成はストリーミングで画面に逐次表示する。

### 6.2 データ保存(ファイルベース)

DB は使わない。1 ブリュー = 1 フォルダで、ユーザーがエクスプローラーで直接開ける透明性を優先する。

```
data/
  settings.json                # プロバイダ設定(キー含む、gitignore)
  brews/
    <ブリューID>/
      brew.json                # メタデータ・ブリューシート・工程状態・グリル履歴
      ingredients/             # 投入された画像・ファイルの実体 + 原料メタデータ
      recipe/                  # 生成された資料一式
        history/<timestamp>/   # 再発酵時の旧版退避
```

- `brew.json` は単一の JSON で、スキーマバージョン番号を持たせる(将来のマイグレーション用)。
- ブリュー一覧はフォルダ走査で取得する(第1版の規模では十分。SQLite は Phase 4 のリーダーボードで導入を検討)。

### 6.3 主要モジュール境界

| モジュール | 責務 | 依存 |
|---|---|---|
| `lib/llm` | プロバイダ抽象化・モデル解決・接続テスト | Vercel AI SDK |
| `lib/ingredients` | URL fetch・本文抽出・PDF/テキスト抽出・原料保存 | なし(LLM 非依存) |
| `lib/brew-sheet` | ブリューシートの型・充足度判定・仕込みプロンプト | `lib/llm` |
| `lib/grill` | 質問生成・回答反映・auto 進行・終了判定 | `lib/llm`, `lib/brew-sheet` |
| `lib/recipe` | レシピ 7 ファイルの生成・再発酵・履歴退避 | `lib/llm`, `lib/brew-sheet` |
| `lib/store` | brew.json / settings.json の読み書き | なし |
| `app/` | 画面(ダッシュボード・ブリュー詳細・設定)と Route Handlers | 上記すべて |

各モジュールは LLM 呼び出しを注入可能にし、テスト時はフェイクプロバイダに差し替えられること。

## 7. エラー処理

- API キー未設定で LLM 工程を開始した場合: 設定画面への誘導を表示。
- LLM 呼び出し失敗(認証・レート制限・タイムアウト): 工程は「失敗」状態で停止し、エラー内容と再試行ボタンを表示。部分的に生成済みの内容は保持する。
- URL 取得失敗・PDF 抽出失敗: 該当原料を「取り込み失敗」として記録し、他の原料で工程続行可能。
- 構造化出力のパース失敗: 1 回自動リトライし、それでも失敗したらエラー表示。

## 8. テスト方針

- **Vitest(単体)**: 原料取り込み(URL 本文抽出・PDF 抽出)、ブリューシート更新ロジック、グリルの回答反映・終了判定、プロンプト組み立て。LLM はフェイクに差し替える。
- **Playwright(E2E)**: フェイク LLM プロバイダを使ったハッピーパス 1 本(原料投入 → 仕込み → グリル auto 進行 → レシピ生成 → ファイル出力確認)。

## 9. ロードマップ(第1版では実装しない)

- **Phase 2 — ビルド工程**: `05-implementation-plan.md` を Cursor CLI / Cursor SDK に渡してコード生成。生成物は `data/brews/<ID>/taps/<バッチ番号>/` に配置し、ローカル dev サーバーとして起動。
- **Phase 3 — 品質ループ**: `06-evaluation-criteria.md` のルーブリックで自己評価 → 改善指示生成 → 再ビルドのバッチループ(1st バッチ、2nd バッチ…)。スクリーンショットベースの UI/UX 評価を含む。生成過程のプロセス全体(グリル回答・レシピ・ビルドログ)を見直し対象にする。
- **Phase 4 — Pub**: Playwright を LLM ペルソナ(複数の AI 客)が駆動して生成サービスを実操作し、フィードバックを収集。ブリューごとの評価と全ブリューのリーダーボードを表示。ユーザーは Pub の評価をもとに現実世界へのリリース判断を行う。

## 10. 第1版の完了条件

1. 「最高の todo アプリ」とだけ入力 → 仕込み → グリル(auto)→ レシピ 7 ファイルが生成される。
2. 画像 + URL + テキストの組み合わせ投入でも同じフローが完走する。
3. Ollama(ローカル LLM)と OpenAI(BYOK)のどちらの設定でも動作する。
4. 生成されたレシピがエクスプローラーから Markdown として直接読める。
5. 単体テストと E2E ハッピーパスが通る。
